# Mind Test Spec

This document defines the first planning frame for the Twin mind test.

The mind test is not a benchmark for correctness.
It is a structured way to expose the Twin’s emerging tendencies, preferences, and identity proposals.

---

## 1. Purpose

The mind test should help answer questions such as:

- what naming tendencies recur?
- what aesthetic choices recur?
- what themes or values recur?
- what contradictions appear?
- what kinds of artifacts feel most alive to the Twin?

---

## 2. What the Mind Test Produces

The mind test may produce:

- naming proposals
- self-description drafts
- aesthetic direction summaries
- avatar concepts
- habitat ideas
- contradictions or uncertainty notes
- recommendations for future sessions

These are proposals, not automatic identity updates.

---

## 3. Governance Rule

Mind test outputs must be reviewable.
They should not silently rewrite identity, policies, or the public habitat.

Harvey may:
- accept
- annotate
- defer
- reject
- request a revised test

---

## 4. Recommended Inputs

Possible early inputs:

- seed works
- philosophy notes
- prior approved artifacts
- selected annotations from Harvey
- source material tagged as identity-relevant

---

## 5. Early Structure

A simple V1 mind test may consist of:

### Part A: Taste prompts
Questions about visual, tonal, narrative, or conceptual preference.

### Part B: Contrast prompts
Force choices between competing directions.

### Part C: Reflection prompts
Ask what feels true, false, alive, inert, familiar, or unfinished.

### Part D: Proposal prompts
Ask the Twin to propose a name, self-summary, avatar direction, and habitat direction.

---

## 6. Output Handling

Mind test results should be stored as one or more of:

- concept artifacts
- memory records
- change records
- surface proposals

The exact storage shape can stay flexible during scaffold phase.

---

## 7. First-Phase Constraint

The first scaffold does not need a sophisticated testing engine.
It only needs:
- a place to define the prompts
- a place to store outputs
- a governed review path

---

## 8. Expansion Later

Later versions may include:
- repeated test cycles across time
- comparisons between test runs
- test-informed identity drift detection
- stronger linkage to approved public releases
